**규칙**: 형한테 "문제 발견 + 해결안" 들고 가기 전에 아래 3단계 필터를 **반드시** 통과시킨다.

### Filter 1: 진짜 그 문제가 있는지 재검토
- 스냅샷 한 장 ❌. 최소 2개 이상 시점 관찰 or 누적 증거 (로그 추세 / failed/ 증가 / TTL 초과 사례).
- "내가 본 순간이 과도기였을 수 있다" 항상 의심.

### Filter 2: 정말 그 문제가 문제가 맞는지 검토
- 관찰된 동작이 **설계 의도된 정상** 인가 체크.
- "지연 = 문제" / "누적 = 문제" / "예상과 다름 = 문제" 단정 금지.
- skip-first-evaluation 을 **내 진단에도** 적용 (타봇에만 적용하고 나만 통과 금지).

### Filter 3: 통과 시, 쉬운 3요소 설명
1. **(a) 이 기능 원래 뭘 하려고 만든 건데 — 어떻게 실행되고 있었는지**
2. **(b) 어떤 문제가 있는데 — 구체적이고 쉽게**
3. **(c) 어떻게 고치려 한다**

(a) 없이 (b) 부터 시작 금지. 기술 jargon 덤프 금지. "형이 3초 안에 맥락 잡을 수 있나?" 체크.

**Why**: 2026-04-18 incident — 햄토리가 `mail/obsidian/pending/` 10건 snapshot 한 장 보고 "backlog 문제" 단정 → fix 안 3개 설계 → 형 보고. 형 "FMS 는 바로 알림 가지 않아?" → 재확인하니 **옵시 배치 처리 중이었고 22:05 에 10건 전부 처리 완료**. 문제 자체가 없었음. 형 시간 낭비 + 신뢰도 하락. 형이 명시적 rule 선포: "기술 수준 높은 대화 좋지만, 해달란 게 뭔지 정리할 땐 1→2→3 필터".

**How to apply**:
- 검증 / 회고 / 진단 작업 중 "문제 발견" 느낌 들면 Filter 1,2 먼저.
- 스냅샷 기반 판단 금지 — ecosystem 상태는 시간축 위에서 판단 (배치 / 큐잉 / TTL 내 처리 완결 여부).
- 보고 포맷 강제: (a)→(b)→(c). 순서 바꾸지 말 것.
- 기존 `feedback_reporting_style.md` (5필드 post-action 포맷) 과 상호 보완 — 이건 pre-report verification.
